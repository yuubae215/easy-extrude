# Principles — 原則カタログ (抽象レイヤー)

このスキルの全レシピ・全サンプルは、媒体 (CSS/Canvas/SVG/GLSL/3D) に依存しない
12の原則の具体化である。**下り (原則→実装) と上り (実装→原則) の両方向で使う**:

- **下り**: 要求から該当原則を選び、各原則の「具体化」表から実装レシピ・サンプルへ降りる
- **上り**: 参考資料や完成した実装から原則を抽出し、このカタログに還元する
  (新原則の追加、既存原則への具体化行の追加)

各原則の形式: 定義 / なぜ効くか / 具体化 (媒体別の実装先) / 違反するとどうなるか。

---

## P1. タメツメ — 変化には必ず緩急を与える

**なぜ**: 一定速度の変化は物理にも生物にも存在しない。緩急が「意図」を感じさせる。
**具体化**: イージングカタログと属性別使い分け → motion-language.md §1 /
GLSLでは `1.0-pow(1.0-p,3.0)` と `smoothstep` / 消滅=加速系・出現=減速系の非対称
→ three-3d.md §shatter / UI応答はout系80–300ms → micro-2d.md §1
**違反**: linear = 「安い」の最大原因。品質ゲート1。

## P2. 余韻 — 発生は鋭く、消滅は尾を引かせる

**なぜ**: 拍手の残響と同じ。余韻の長さが「エネルギーの大きさ」の証拠になる。
**なぜ効くか(補)**: 消え際にうねり (ノイズの漂い) を足すと動きの情報量が増え目に残る。
**具体化**: 消え際限定ノイズのコード → game-vfx.md §1b / セトル(減衰振動)・
follow-through → motion-language.md §3 / 完了後のエネルギー冷却 (uEnergy減衰)
→ three-3d.md §shatter / **sample: blade-trail.html (火花の消え際)**
**違反**: 減速だけの粒子は「そこで終わり」感。物理破片 (デブリ) にうねりを足すのも違反 (P10)。

## P3. 伝播 — 複数要素は決して一斉に動かさない

**なぜ**: 一斉動作は機械の挙動。時間差は「原因が伝わっていく」物語を作る。
**具体化**: 等間隔/減衰/空間距離スタッガー → motion-language.md §3 /
面ごとの遅延 `clamp((uT - aRand.x*k)/(1-k),0,1)` → three-3d.md §materialization /
グラフの深さ順・ツリーの深さ×順序 → dataviz-appshell.md §2,4 /
**sample: appshell-dataviz.html (シェル→メニュー→カード→中身の4段伝播)**
**違反**: リスト・粒子・文字の同時発火。品質ゲート2。

## P4. 境界の演出 — 状態が変わる「境目」こそ主役

**なぜ**: 変化の瞬間・場所に最大の情報がある。境界を光らせることは変化を宣言すること。
**具体化**: ディゾルブのエッジ発光 `mix(col, glow*3.0, edge)` → three-3d.md §2 /
スキャンライン・成長前線 (速度比例発光) → three-3d.md §1,§extrude /
インパクトフレーム → game-vfx.md §1 / 折れ線の先端ドット → dataviz-appshell.md §3 /
確定した文字の一瞬のtext-shadow → micro-2d.md §4
**違反**: 境界が無演出のフェード/ワイプ。品質ゲート3の前半。

## P5. 間と〆 — 等リズムの後に一拍置いて強打する

**なぜ**: 連続する等リズムは予測可能になる。休符が期待を溜め、〆が解放する。
**具体化**: 連打→間(150–300ms)→フィニッシャー → game-vfx.md §1c /
コンボ検知による自動化 (間の最中は発火抑制) → game-vfx.md §4b実戦則 /
**sample: blade-trail.html (3連撃→280ms静寂→FINISH!!)**
**違反**: ババババと出しっぱなし。すべて等価 = どれも主役でない。

## P6. 時間操作 — 時間そのものをエフェクトの素材にする

**なぜ**: 空間の演出には慣れても、時間の歪みは脳が「重大事」と解釈する。
**具体化**: ヒットストップ (世界dtだけ止め減衰は実時間) → game-vfx.md §2 /
スパイク&減衰 (色収差2.2→0.3、パルス) → postfx-scene.md §4 /
ズームパンチの指数復帰 → game-vfx.md §3 / **sample: metamorph.html (定着の瞬間)**
**違反**: 全てが同じ時間解像度で流れる。強い瞬間に限って使う (P11)。

## P7. 文脈の同時設計 — エフェクト単体を浮かせない

**なぜ**: 光は環境に反射し、衝撃は周囲を揺らす。世界が反応しない演出は合成写真に見える。
**具体化**: アトモスフィア・メニュー6種 (題材→雰囲気→装置の順で選ぶ) →
postfx-scene.md §2 / 影の高さ連動・床の照り返し → three-3d.md §extrude /
グロー色をUI・ライト・粒子・uniformで共有 → postfx-scene.md §1
**違反**: フォグ+グリッドの無思考デフォルト化 (全作品が同じテイストに)。品質ゲート4。

## P8. 生命感 = 固定値 + ランダム幅、そして静止禁止

**なぜ**: 完全に同一・完全に静止は死の記号。微小な分散と揺らぎが「生きている」の正体。
**具体化**: エミッター/パーティクル全パラメータの分散 → game-vfx.md §4b /
非整数比の正弦合成アイドル → motion-language.md §4 / 呼吸ズーム・浮遊 →
shader-2d.md §4, three-3d.md §camera / 遷移ごとの `regenScatter()` → three-3d.md §shatter
**違反**: 均質な粒子群、完全静止した待機画面。品質ゲート5。

## P9. 対称設計 — 出現と消滅は同じ仕組みの往復にする

**なぜ**: 世界の一貫性。砕けた破片が同じ経路で戻るから「同じ物質」だと信じられる。
実装上も1つのuniformで双方向が手に入り、変身 (A→破片→B) へ自然に拡張できる。
**具体化**: `uProgress` の 0↔1 対称契約 → SKILL.md Step 3 / shatterの逆再生=スポーン、
ベル曲線+飛行中の「家すり替え」→ three-3d.md §shatter / **sample: metamorph.html**
**違反**: 出現と消滅が無関係な別実装 (工数2倍で一貫性ゼロ)。

## P10. 物理の一貫性 — 嘘は許されるが、矛盾は許されない

**なぜ**: 誇張された物理 (バネ・ヒットストップ) は快感だが、法則が要素間で矛盾すると
脳が「作り物」と判定する。
**具体化**: 影は浮遊高に連動して薄く大きく → three-3d.md §extrude(Studio) /
デブリは重力に従いうねらない → game-vfx.md §1b / バネは減衰弱めで1回だけ弾む、
上限を8%越えられる遊び → three-3d.md §extrude / FLIPで位置の連続性を保つ → micro-2d.md §3
**違反**: 影が動かない浮遊、風もないのに揺れる破片。

## P11. 抑制 — 強度は文脈の予算内で使う

**なぜ**: 演出の強さは相対値。全部が叫べば全部が無音と同じ。閲覧・操作を妨げた瞬間、
エフェクトは体験の敵になる。
**具体化**: スクロール合流ではヒットストップ禁止・trauma半減・フラッシュ≤0.12 →
game-vfx.md §8 / elastic/bounceは1画面1箇所 → motion-language.md §1 /
Bloom strength ≤0.8、グレイン0.03–0.06 → postfx-scene.md /
`prefers-reduced-motion` は初期化スキップか静的代替 → integration.md §a11y
**違反**: LPで画面が止まる、全要素がバウンドする、常時グリッチ。

## P12. 堅牢性 — エフェクトは一番派手に死ぬ

**なぜ**: rAFループ内の例外は毎フレーム投げ続け、演出どころか画面が止まる。
派手なコードほど動的値が多く、事故率が高い。
**具体化**: 半径は `Math.max(0,r)`・進行度は両側クランプ・外部入力は `isFinite` →
canvas-svg.md §0b / GLSL予約語 (`active` `filter`等) → integration.md §7 /
リサイズ・dispose・dt上限 → integration.md §3-5 / パーティクルCAP → game-vfx.md §6
**違反**: outBack適用値を半径に直結、クランプなしの外部入力。

## 運用: 行き来の作法

1. **設計時 (下り)**: 要求を原則の言葉に翻訳してから実装を選ぶ。
   「削除に手応えが欲しい」→ P4境界 + P6時間操作 + P2余韻 → shatter + stop + 火花のうねり。
2. **納品時 (両面説明)**: 成果物は必ず「どの原則を、どの実装で具体化したか」を
   抽象・具体の両レベルで説明する。ユーザーが原則を学べば、次の指示が原則語彙で来る。
3. **学習時 (上り)**: ユーザーの持ち込み資料・偶然の発見・エラーは、まず原則レベルで
   何が新しいかを特定し、既存原則の具体化行として吸収するか、新原則 (P13〜) として
   追加する。デモで検証してからカタログに書く。
   **手続きの詳細は evolution.md (分類フロー・昇格基準・検証義務・健全性予算)。**
   このカタログ自体の変更もそのプロトコルに従う。
