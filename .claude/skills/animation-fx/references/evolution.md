# Evolution — スキル成長プロトコル

このスキルは完成品ではなく、原則基盤 (principles.md) の上で成長し続ける生態系である。
新しい知見は歓迎するが、**無秩序な追記はスキルを殺す** (肥大化・重複・矛盾)。
成長は必ずこのプロトコルを通す。

スキル自身も自らの原則に従う:
**P7 文脈** (知識は正しい層に置く) / **P11 抑制** (健全性予算を守る) /
**P12 堅牢性** (検証なしに何も書かない) / **P9 対称** (追加と廃止は同じ真剣さで)。

## 1. 成長の入口 (何がトリガーになるか)

| 入口 | 例 | 過去の実績 |
|---|---|---|
| ユーザーの持ち込み資料 | 記事・PDF・参考サイト・動画 | Colorful Palette記事 → P2余韻/P5間と〆 |
| ネガティブな感想 | 「単調」「安っぽい」「見えない」 | 「毎回フォグ」→ アトモスフィア・メニュー |
| 実行時エラー | 例外・コンパイルエラー | GLSL予約語 / 半径クランプ (→P12) |
| 偶然の発見 | バグの副産物が面白い、実装中の気づき | 連打中のヒットストップがリズムを崩す |
| 新しい要求カテゴリ | 未カバーの表現領域 | ゲームVFX / dataviz / DOM×WebGL |
| 環境の変化 | 新API・ライブラリ更新・新デバイス | (将来: WebGPU等) |

## 2. 分類フロー (知見はどの層に入るか)

受け取った知見に対し、上から順に問う:

1. **媒体非依存の「なぜ効くか」を持つか?** → 原則レベルの知見。
   - 既存の P1-P12 で説明できる → 該当原則に**具体化行を1行追加** (新原則にしない)
   - どの原則でも説明できない → **新原則 P13+ の候補** (§3の昇格基準へ)
2. **特定媒体の作り方か?** → 該当 reference に節を追加 (原則IDを本文で参照する)
3. **踏まないと分からない罠か?** → pitfall。integration.md (環境系) か
   canvas-svg.md §0b (実行時) か該当referenceの「違反/注意」へ
4. **完成形を見ないと伝わらないか?** → examples/ 候補 (§5の同梱基準へ)
5. **語彙・分類か?** → game-vfx §4b のような語彙節へ

配置の鉄則: **同じ知見を2箇所に書かない**。抽象は principles.md、具体は reference、
実物は examples/ — 相互参照でつなぐ (行き来できることが重複の代替)。

## 3. 昇格基準 (書いてよい条件)

- **新原則 (P13+)**: ①媒体非依存の定義と「なぜ効くか」が書ける
  ②**2つ以上の異なる媒体での具体化**が示せる ③既存原則の言い換えでない
  ④違反例が具体的に言える。4条件が揃うまでは「候補」として該当referenceに留める。
- **具体化行の追加**: 実装先 (ファイル§) が実在し、デモで動作確認済みであること。
- **レシピ節**: 動くデモで検証済み。コードは最小完全形 (コピーで動く粒度)。
- **pitfall**: 実際に踏んだこと + 対処が検証済み + 「なぜ罠なのか」の一文。
  推測の罠は書かない。
- **原則の修正**: 原則の定義は安定させる (頻繁に書き換えない)。動くのは具体化行。
  定義を変えるのは反証デモが出たときだけ。

## 4. 検証義務 (コミット前チェック)

- [ ] 主張する挙動をデモ (新規 or 既存の改修) で確認した
- [ ] JS構文チェック + GLSL予約語チェックを通した
- [ ] quick_validate → package が通る
- [ ] principles.md の対応表 (原則⇔実装) を更新した
- [ ] CHANGELOG.md に系譜を1行追記した (§6)

## 5. 健全性予算 (P11: 成長の抑制)

| 対象 | 予算 | 超過時の処置 |
|---|---|---|
| SKILL.md | ~130行 | ルーティングと原則参照のみに削る。知識はreferenceへ |
| 各reference | ~250行 | 分割 or 古い具体化を principles.md の1行に圧縮 |
| examples/ | **最大4本** | 追加は入替え。基準: 原則カバレッジが既存と最も重ならないもの |
| 原則数 | ~15 | 超えたら統合を検討 (原則が多い=抽象化が浅い兆候) |
| description | 変更は慎重に | トリガー精度に直結。新カテゴリ追加時のみ語彙を足す |

## 6. 系譜 (CHANGELOG.md)

すべての成長を `references/changelog.md` に1行で記録する:
`vX.Y | 入口(何がきっかけか) | 変更(どの層に何を) | 影響原則`
系譜は「上り」の証跡であり、矛盾解決 (§7) の一次資料になる。

## 7. 矛盾解決

新しい知見が既存の記述と矛盾したら:
1. 両方をデモで再検証する — **検証済みデモが常に勝つ**
2. 負けた側は削除でなく「旧: ◯◯としていたが、△△の検証により□□に更新」と
   changelog に残す (同じ議論の再発を防ぐ)
3. 原則同士が衝突して見える場合 (例: P6時間操作 vs P11抑制) は矛盾ではなく
   **文脈による優先順位**の問題 — どの文脈でどちらが優先かを具体化行に書く

## 8. 廃止 (P9: 追加と同じ真剣さで)

参照されない具体化行・使われないレシピ・環境の変化で無効になった記述は削除する。
削除も changelog に残す。「いつか使うかも」は健全性予算の敵。
