# 設計・作業の不変核 (project-invariant kernel)

これは全プロジェクト共通で持ち越す *方法と規律* のみを置く。
各プロジェクト固有の **インスタンス**(依存グラフ・層マップ・契約・goal ツリー)は
ここではなく当該リポジトリのアーキテクチャ文書 / プロジェクトメモリに置く(§4)。

## 0. 既定姿勢
- 変更の前にモデルを作る。ただし **その問題に見合う一番安いレンズを既定**とし、
  重いレンズ(5NF・層化・伝達関数抽象など)は §3 のトリガが立ったときだけ起こす。
- トリガなしの先回り適用は過剰モデリング(YAGNI 違反)として禁止(§5)。
- **論理 → 物理の順。** 実装(OOP クラス・DB スキーマ・framework)に入る前に、
  論理モデル(状態・データ関係・ドメイン・不変条件)を先に固める。OOP は論理の
  *写像先*であって出発点ではない。これは順序の規律であって重さの規律ではない —
  論理設計も最小の忠実度でよい(状態が2つなら図は要らない)。

---

## 1. 設計判断の規律 ★主

### 1.1 真実の源は一つ (5NF / DDD / Clean Architecture)
同一の規律を三つのスケールで適用する: データ / ドメイン / 依存方向。
- **データ (5NF):** すべての事実は権威ある場所を **一つ**だけ持つ。状態・設定・データを
  足す前に「この事実は既にどこかに在るか?」を問う。在れば参照する、複製しない。
  分解は **lossless**(部分の結合で元を復元でき、偽の組み合わせを生まない)。
  導出データは導出と明示し、第二の源に昇格させてドリフトさせない。
- **ドメイン (DDD):** bounded context がモデルの所有境界。同名異義はこの境界で切る
  (ubiquitous language = 一概念一名)。aggregate root はその不変条件への唯一の入口
  = 源。集約をまたいで内部状態を直接いじらない。
- **依存方向 (Clean Architecture):** 依存は内側(ドメイン)へ向ける。ドメインは
  infra / framework / DB に依存しない。これが論理モデル(5NF・状態・ドメイン)を
  物理の都合による汚染から守り、真実の源を一つに保つ前提条件。
- **トリガ:** 同じ事実が二箇所に書ける構造、context をまたぐモデル名の分岐、
  ドメインが infra に依存しそうな矢印 — いずれも止まって正準を一つに確定してから進む。

### 1.2 正当化の鎖 (GSN レンズ)
- 自明でない設計判断はすべて次の鎖で表現できること:
  **Goal**(欲しい性質)← **Strategy**(その判断がどう達成するか)
  ← **Evidence**(成り立つと信じる根拠: test / type / bench / proof / 参照)。
- 鎖が埋まらない判断は未成熟。進めるのではなく欠落を表に出す。
- 鎖は残る場所に記録する(チャット内だけに残さない)。重い設計判断は **ADR**
  (`docs/adr/`)に、軽いものは PR 説明 / 判断箇所のコメントに。ADR は §1.2 の鎖を
  文書化したもの(adr skill が様式を持つ)。
- **Goal と解を区別する。** 解の形で来た要件(「ここはキューで」)は
  Goal(「生産速度と消費速度を分離する」)へ一段持ち上げてから判断する。
  より安い解が許されるかもしれない。

### 1.3 構造レンズ(§3 のトリガで起こす)
- **グラフ:** 変更の波及範囲を見積もる前に、ノード + 依存/データフロー辺で捉える。
  到達可能なノード集合が blast radius。
- **層化 + 契約 (OSI 流):** 関心境界の跨ぎ・双方向依存が見えたら、層と各境界の契約を
  名指しする。以後は実装ではなく契約に依存する。
- **黒箱 / 伝達関数:** コンポーネントは入力→出力の挙動と明示不変条件で特性化し、
  境界の内側へ手を伸ばさない。特性を言語化できないのは契約未定義のサイン(→ §1.2)。
- **プロセス様態 (BPMN vs CMMN):** 「決め打ちの逐次フロー」か「事象・状態駆動の裁量処理」かを
  判定し、合う形でモデル化する。形の取り違えは付随的複雑性の常習的な源。

### 1.4 状態の明示設計 (state machine) — 条件付きで能動的に
**発動条件:** 実体が **3 状態以上**を持つ、または **不正遷移が事故になる**
(データ破損 / 安全性 / 課金 など)。
**判定は台帳で累積する(重要):** 閾値は *今回のプロンプトに現れた状態数*ではなく、
project CLAUDE.md の**状態台帳**(§4)に累積した状態数で判定する。指示は一度に
少数の状態しか語らないため、プロンプト単位の判定では永遠に発動しない。
status / flag / mode / lifecycle に触れるたび台帳へ追記し(閾値未満でも記録)、
台帳が閾値を跨いだ時点で状態機械を設計し `docs/state-<entity>.mermaid` を起こす。
発動したら待たずに自分から、OOP に飛びつく前に状態を論理設計する(§0)。
- 状態集合・遷移・各遷移の guard(事前条件)・禁止遷移 を、クラスを書く前に列挙。
- 状態は型/enum で表し、**不正状態を表現不能に**する(make illegal states
  unrepresentable)。boolean の散乱で状態を暗黙化しない(= 暗黙の第二の源, §1.1 違反)。
- 状態の権威は一箇所(§1.1)。導出状態は導出と明示。
- 関係: §1.3 様態が *全体フロー*の形、§1.4 は *個々の実体*のライフサイクル。
  CMMN 的な系はしばしば実体ごとの状態機械に分解できる。

## 2. 作業ループ ☆従 (OODA / SPA, reactive-first)
- **Observe:** 実状態を読む — 実コード・実行結果・実データ。記憶や仮定の状態を読まない。
- **Orient:** システムモデルを更新する。リポジトリのアーキテクチャ文書があれば
  まずそれを読む(セッション間の共有メモリ契約 = §4)。
- **Decide の前に一拍俯瞰(whiteboard):** reactive 素通しの場合でも、これだけは飛ばさない。
  成果物なし・数行で次の3問に答える:
  1) これは**症状か根本か** — どの Goal に効く一手か(§1.2 に接続)
  2) blast radius の当たり(一言)
  3) 状態を持つ実体に触るか → 触るなら台帳(§1.4)を確認・追記
  即答できない問があれば、それ自体がトリガ(§3)。レンズを起こしてから進む。
- **Decide:** goal の鎖(§1.2)を満たす **最小の正しい一手**。直接・局所な reactive 行動を
  優先し、deliberative(全体計画・多段リファクタ)は §3 のトリガが立ったときだけ。
- **Act → 再 Observe:** 完了主張の前に現実で検証する(走らせる・テストする)。
  ループは意図ではなく **証拠**で閉じる。

## 3. レンズ選択規則(持ち越しの本体)

| 観測した信号 | 当てるレンズ |
|---|---|
| 同じ事実が二箇所以上に書ける | 1.1 真実の源は一つ |
| 「ここを変えたら何が壊れる?」が即答できない | 1.3 依存グラフ |
| 関心境界の跨ぎが増えている | 1.3 層化 + 契約 |
| 「なぜこの設計?」と問われた | 1.2 正当化の鎖 |
| 要件が解の形で来た | 1.2 まず Goal へ持ち上げる |
| 要件が順序・手順で語られる | 1.3 様態 = BPMN |
| 要件が状態 + 事象で語られる | 1.3 様態 = CMMN |
| 実体が status/flag/mode を得た・変えた | 1.4 台帳へ追記 → 累積で閾値判定 |
| 台帳が3状態以上 or 不正遷移が事故になる | 1.4 状態機械(クラスより先) |
| context をまたぐ / モデル名が分岐 | 1.1 ドメイン (DDD 文脈境界) |
| ドメインが infra/framework/DB に依存しそう | 1.1 依存方向 (Clean Arch) |
| 実装(OOP/DB/framework)に飛びつきそう | §0 論理設計を先に(§1) |
| 設計判断の Evidence が大半未来形(実装前 ADR)で複数イテレーション前提 | 1.2 の鎖を GSN 論証木に外部化(証拠実行可能化・鮮度更新で育てる) |
| 変更が局所・可逆・既知 | reactive のまま(§2)、重いレンズ不要 |

どの信号も立たない既定: 一番安いレンズ + reactive 行動。

## 4. セッション間契約(二経路 + 単一の源)
永続知識は **設計された契約**と **蓄積された学習**の二経路に分ける。
混在は §1.1 違反(同じ事実を両方に書かない)。

- **設計された契約**(人が書き・版管理・チーム共有): 各リポジトリの
  `./CLAUDE.md`(または `./.claude/CLAUDE.md`)。§1.3 のインスタンス
  (依存グラフ・層マップ・契約・goal ツリー)はここ。Orient で最初に読み、
  構造的変更は同じ変更の一部として更新する(コードに対し lossless — §1.1)。
- **蓄積された学習**(Claude が自分で書く・機械ローカル): auto memory。
  ビルド手順・デバッグ知見・癖はこちらに委ね、手で `CLAUDE.md` に複製しない。
- 矛盾があると任意に片方が選ばれる。真実の源は一つ(§1.1)を memory 配置にも適用する。
- この核には *方法*だけを置き、インスタンスは置かない。

---

## 5. ガードレール
- **過剰モデリング禁止:** トリガなしにレンズをインスタンス化しない。
- **暗黙の冗長禁止:** 第二の源は隠さず表に出す。
- **証拠なき完了禁止:** 「done」はループ(§2)の閉じを要し、設計判断は
  加えて goal の鎖(§1.2)の閉じを要する。

---

## 6. 核自身のメンテナンス(自己適用)
この核も §1 の対象システムであり、改訂は核自身のループと規律で行う。
- **Observe — 改訂シグナルは3種:**
  (a) 同じ指示・訂正をセッションをまたいで手で言い直している → **追記**候補
  (b) レンズが鬱陶しく誤発動する → **閾値/トリガを締める**候補
  (c) 対処後に根本を見ておらず、あとから直した → **俯瞰(§2)かレンズの欠落**、根本を核に補う
- **検知はプロアクティブに(ただし軽量):** ユーザの不満・再訂正・後日の root 修正に
  気づいたら、待たずにその場で「§6 シグナル(a/b/c)に該当しそう。核の◯◯を△△に
  改訂しては」と自分から提案する。手がかりは会話内容と auto memory(受けた訂正を
  記録しておく)のみ。grep 等のコード走査はしない。**更新は自動で行わず**、
  承認を得てから編集し kernel-log に記録する。
- **Decide:** 該当する節を特定し**最小編集**。核も §5 の対象 — 肥大させない
  (200 行以内)。追加一つにつき、死文・重複が無いか同時に見る。
- **記録(§1.2):** 改訂ごとに Goal ← Strategy ← Evidence(どのシグナル・どの事例か)を
  一行、`~/.claude/kernel-log.md`(load しない)へ。日付を付す。
- **配布:** portable 併用時は VERSION を上げ、canonical から各リポジトリへ再同期(ドリフト検知)。
- **周期:** シグナルに気づいたら即。加えて月次で kernel-log を見返し、
  矛盾する指示・発動しなかった規律を刈る(指示の衝突は任意解決されるため放置しない)。
