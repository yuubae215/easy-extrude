# ADR-074: BFF <-> コアAPI の I/O 契約

- Status: Accepted (§6 契約型の置き場所のみ ADR-082 で改訂 — 外部 repo submodule →
  repo 内 `packages/grasp-contract`。意味論・contractVersion ガード・導出規律は有効なまま)
- Date: 2026-06-21
- 関連 (追記): ADR-082 (置き場所の吸収 + 契約壁 CI ガード)

## Context

判定エンジン (コア, レイヤ B) を独立したコアAPI サーバに置き、BFF から
委譲呼び出しする構成を採る。BFF とコアAPI は
**別プロセス・別デプロイ**になるため、両者の境界 = ネットワーク越しの I/O 契約を
明文化し、片側だけ古いままデプロイされたときの事故を実行時に弾けるようにする。

## Decision

### 1. 入力 (BFF -> コアAPI)

- `graspSearch` 宣言。前提とする Layout DSL のスキーマバージョン (`version: "layout/x.y"`)
  を明記して渡す。
- 契約バージョン `contractVersion` (整数) を含める。

### 2. 出力 (コアAPI -> BFF)

- **上位N件**の姿勢ランキング (最良1件のみではない)。
- 各候補に **スコア内訳** を付ける:
  - `ik_solvable` 可否 (IK 可解性)
  - 干渉 (interference) 判定
  - リーチ (reach) 判定
  - 各 objective の **正規化済み (0-1) 値**
  - 加重和の総合スコア
- 上位N件 + 内訳にする理由:
  - BFF 側で「1位がダメだった時のフォールバック」が作れる。
  - テンプレ (2a) の「なぜこの姿勢が選ばれたか」の可視化 = 説得材料になり、商品価値に直結。
  - 最良1件だけだと上の両方が作れない。

### 3. 契約バージョン `contractVersion`

- 内部用なので素朴な整数 (`contractVersion: 3` のような)。public スキーマのような
  prefix 付き semver にはしない。
- **上げる条件 (ゆるい規約)**:
  - 必須フィールドの追加・意味変更 -> 上げる。
  - 任意フィールドの追加 -> 上げない。
- 両側 (BFF / コアAPI) を自分が同時に直せるので、破壊変更そのものは気軽でよい。
  恐れて我慢しない。

### 4. 不一致時の挙動

- コアAPI は想定外の `contractVersion` を受けたら **400 で即拒否**。曖昧に処理しない。
- public スキーマでの「version 強制チェック」と同じ規律を、内部契約にも適用する。
- 狙い: コアAPI を先に更新して BFF が古い (または逆) のズレを、デバッグ一瞬で特定できる。

### 5. public スキーマとの関係

- 契約は public スキーマの `version` を *参照* するが、契約自身の番号 (`contractVersion`)
  とは **別系統**(スキーマ `version` は外部互換の約束・破壊変更は慎重に、`contractVersion`
  は BFF <-> コアAPI 間の内部取り決め・両側を同時に直せるので気軽に上げてよい)。

### 6. 契約型の置き場所

- 契約の型 (入出力 + `contractVersion`) は、BFF にもコアAPI にも属さない **中立な共有
  スキーマパッケージ**に置き、両者が import する。
- コピペで両側に型を持つと、片方だけ直して不一致 = まさに `contractVersion` で検知したかった
  事故を型定義レベルで起こす。だから「契約だけ共有」の実装は「契約を共有パッケージに置いて
  両者が参照」になる。
- 移行タイミングは段階的でよい。この ADR は **契約の意味論を確定**するもので、
  物理的な置き場所の移行完了を待たない。
- 実装状況 (2026-06): 中立パッケージ `packages/grasp-contract/` を新設。言語非依存の
  JSON Schema を正本とし、コア側の pydantic がその binding (準拠テストあり)。
  消費側が BFF=TS / コア=Python と分かれるため、単一言語パッケージではなく JSON Schema
  を中立正本にした。

## Consequences

- BFF はコアの *中身* (段階0 総当たりか将来の最適化か) を一切知らずに更新できる。
  BFF が知るのは「入力の形 / 出力の形 / `contractVersion` / エラーの約束」の4つだけ。
- コアの実装が進化しても BFF は無変更でいられる (疎結合の狙い)。
- 契約の最小実装ガード: コアAPI の最初の1エンドポイントから `contractVersion` 検証を
  入れる (「後でセキュアに」は手遅れになりやすい)。

## Open (この ADR では確定しない)

- 共有スキーマパッケージを「今作るか / 契約型だけ先行 / DSL submodule は当面据え置き」かの
  移行タイミング (運用判断)。
- コアAPI のホスティング先と、BFF だけが通れる内部認証の具体方式。
