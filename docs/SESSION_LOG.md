- **2026-05-16**: Bugfix — ADR-040 completion: `_objCtrlDrag` primary triple violation. The Ctrl+drag rotate path in `AppController._onPointerMove` was the last ADR-040 violation: it directly mutated `vertices[i].position` via inline quaternion math, bypassing the public API and leaving `_position`, `orientation`, and `localCorners` stale after the drag. Subsequent G-key grab would teleport the Solid (stale `_position` fed into `move(segStartPos, delta)`); subsequent R-key rotate would use the wrong pivot; face extrude would compute wrong local offsets. Fix: at ctrl+drag start, snapshot `obj.orientation` and `obj._position` (instead of corners); in pointermove for Solid, call `obj.rotate(startOrientation, startPos, centroid, quat)` — snapshot-based, zero accumulation error, primary triple consistent after every frame. Non-Solid entities retain the old corner-mutation path (they have no primary triple). No command push (ctrl+drag is intentionally non-undoable, consistent with existing `_objDragging` design). CODE_CONTRACTS §Solid Pose Mutation Must Use Public API already covered this rule; no new rule added (same root value as PHILOSOPHY #1).
