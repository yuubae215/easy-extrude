# 🔒 Concurrency & State Synchronization Strategy

本ドキュメントは、`easy-extrude` における状態管理、非同期処理、および複数プロセス間のデータ競合（Concurrency）に対するアーキテクチャ方針を定義します。

当プロジェクトの並行処理モデルは、**RTOS（リアルタイムOS）における「割り込み処理（ISR）とバックグラウンドタスクの分離」の哲学**を、Webブラウザのシングルスレッド（イベントループ）に適用したものです。

---

## 1. コア・メタモデル：描画ループは「ISR」である

ブラウザにおけるUI入力イベント（Pointer Events）やThree.jsのレンダリングループ（`requestAnimationFrame`）は、**厳格なデッドライン（16.6ms）を持つハードリアルタイムタスク（ISR）**と見なします。

- **絶対原則**: メインスレッド（ISR）をブロックしてはならない。
- リアルタイム性の要求度合い（＝ISR内で処理すべきか、バックグラウンドに回すべきか）に応じて、**楽観的（Optimistic）**と**悲観的（Pessimistic）**のロック戦略を厳密に使い分けます。

---

## 2. 戦略A：楽観的ロック (Optimistic Locking) — リアルタイム・ノンブロッキング

UIの即応性が求められる操作では、ミューテックス的なロック（処理の待機）を一切排除し、止まらないことを最優先とします。

**適用シナリオ:**
- 3D空間でのオブジェクトのドラッグ移動（Grab）
- カメラのオービット操作（OrbitControls）
- 頂点・面などの高頻度な選択状態（`editSelection`）の更新

**RTOS的メンタルモデル:**
センサーデータのリングバッファ読み書きと同様。ロックフリーで常に最新の値を優先し、古いデータによる競合は破棄する。

**実装方針:**
- **Local First**: ユーザーの入力を即座に View およびローカルの Model に反映させる。処理完了や他ノードとの同期を待たない。
- **Last Write Wins / Versioning**: 状態の確定（マウスアップ等）時に `SceneModel` 等の集約ルートへコミットする。競合が発生した場合は最新タイムスタンプ/バージョンを優先し、古い更新はサイレントに破棄またはマージする。
- **禁止事項**: 高頻度イベントのコールバック内で、重い同期処理・Promise の直列待機・UIをフリーズさせるロックを絶対に実装してはならない。

---

## 3. 戦略B：悲観的ロック (Pessimistic Locking) — 整合性重視・ブロッキング

データの不整合がシステム全体の破綻（ジオメトリの崩壊など）を招く複雑なトランザクションでは、リアルタイム性をあえて切り捨て、完全なロックを行います。

**適用シナリオ:**
- 複雑なメッシュの生成、ブーリアン演算など、計算コストが高い処理
- シーン全体のエクスポートや外部ストレージへの保存・ロード処理
- 複数のドメインエンティティにまたがる不可分な一括更新

**RTOS的メンタルモデル:**
メインループにおける低優先度だがクリティカルな設定変更タスク。セマフォやミューテックスでリソースを排他制御し、他のタスクの介入を許さない。

**実装方針:**
- **明示的なフラグ**: 処理を開始する前に、対象エンティティや `SceneModel` に対して明示的なロックフラグ（例: `isProcessing = true`）を立てる。
- **UIのディスエーブル**: ロック中は対象オブジェクトへのポインターイベントを無効化し、ローディング表示でユーザーの介入を拒否していることを視覚的にフィードバックする。
- **確実な解放**: `try...finally` ブロックを必ず使用し、例外発生時でもロックが確実に解除（`isProcessing = false`）されることを保証する（デッドロックの防止）。

```js
// 悲観的ロックの典型実装パターン
async function heavyOperation(entity) {
  entity.isProcessing = true
  try {
    await doHeavyWork(entity)
  } finally {
    entity.isProcessing = false
  }
}
```

---

## 4. アーキテクチャ層への適用と Claude Code への指示

この戦略は MVC/DDD アーキテクチャに以下の通りマッピングされます。新規機能や非同期処理を実装・修正する際は、このルールを必ず順守すること。

| 層 | 役割 |
|----|------|
| **Domain** | ロックの概念を持ち込まない。純粋なエンティティ状態のみ。 |
| **Service / Model** | ロック制御の主体。楽観的競合解決（バージョン比較）と悲観的フラグ管理をここで行う。 |
| **Controller** | Service 層からの例外（ロック取得失敗等）をキャッチし、View の振る舞いに変換する。 |
| **View** | `isProcessing` 等のロック状態を監視し、入力遮断とフィードバック（スピナー等）に専念する。 |

**判断チェックリスト（新しい非同期処理を書く前に確認）:**

1. この操作は 16ms 以内に完了するか?
   - Yes → 楽観的ロック。ブロッキングなし。
   - No → バックグラウンドへ。悲観的ロックで入力を保護する。
2. 失敗した場合、シーンの整合性は保たれるか?
   - Yes → `try/catch` で回復。ユーザーにトースト通知。
   - No → 必ず `try/finally` でロック解放。部分更新を防ぐためトランザクション的に実装する。
3. 他のポインターイベントと競合するか?
   - Yes → `isProcessing` ガードを `_onPointerDown` に追加し、Controller で早期リターンする。
